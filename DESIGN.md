# SJ Spatial Compression — 설계 문서

## 1. 설계 철학

### "압축기가 아니라, OS 데이터 구조화 엔진이다"

전통적 압축기(gzip, zstd)는 패턴 반복을 탐지하여 축소한다.
SJ Spatial은 데이터를 **512×512 고정 이미지 위에 재배열**한다.

핵심 인사이트:
- 집을 그린 512×512 이미지 = 도심숲을 그린 512×512 이미지 = **같은 파일 크기**
- 이미지 크기는 내용이 아닌 **활성 픽셀 수**에만 의존
- 따라서: 원본이 1MB든 1GB든, 같은 활성 패턴이면 이미지 크기는 동일
- **압축률은 보너스. 본질은 OS 실행을 위한 데이터 재배열.**

### "밝기값 상태 자체가 원본을 구성할 정보의 전부"

- Y축 = 시간축이므로, 별도의 이벤트 로그(WH)가 필요 없다
- 각 셀의 밝기값 = 해당 시간(Y)에 해당 바이트(X)가 나타난 횟수
- 이 횟수의 분포 = 원본 데이터의 완전한 공간적 표현

## 2. 인코딩 알고리즘

```
입력: data[0..N-1]  (임의 바이트 스트림)
출력: 512×512 brightness map

for p = 0 to N-1:
    page   = p / 256
    offset = p % 256

    if offset < 128:           // R 걸음
        y = (page * 128 + offset) % 512
        x = data[p]            // 0-255, R존
    else:                      // G 걸음
        y = (page * 128 + (offset-128)) % 512
        x = 256 + data[p]     // 256-511, G존

    canvas[y][x] += 1
```

### 특성
- **RG 걷기**: 2바이트씩 (R걸음 + G걸음), 128쌍 = 1페이지(256B)
- **B값 페이지**: 4페이지(1024B)가 1 Y-사이클(512행)을 채움
- **순환 누산**: Y가 512에서 wrap → 밝기 값이 누적
- **결정론적**: 같은 입력 → 항상 같은 캔버스 (DK 규칙)

## 3. 출력 형식

```
[SJ3F 헤더: 24바이트]
  magic:          "SJ3F" (4B)
  original_bytes: uint64 (8B)
  max_brightness: uint64 (8B)
  active_cells:   uint32 (4B)

[활성 셀 배열: active_cells × (4 + varint)바이트]
  y:     uint16 (2B)
  x:     uint16 (2B)
  delta: varint (1-10B)  = max_brightness - brightness
```

### 델타 압축
- `max_brightness`를 기준으로 각 셀의 delta를 저장
- 최대 밝기 셀: delta = 0 → varint 1바이트
- 대부분의 셀이 비슷한 밝기 → delta 값이 작음 → varint 효율적

## 4. 디코딩 알고리즘

```
입력: 512×512 brightness map + original_bytes
출력: data[0..N-1]

for p = 0 to N-1:
    (y, zone) = 인코딩과 동일한 좌표 계산

    best_x = argmax(canvas[y][zone_start:zone_end])
    data[p] = best_x - zone_start

    canvas[y][best_x] -= 1   // 밝기 소진
```

### 복원 특성
- **주기적 데이터** (바이너리, OS 실행파일): 100% 무손실
  - 1024바이트 주기 = 1 Y-사이클 → 모든 행에서 단일 최대 열
- **비주기적 데이터** (텍스트, 랜덤): 최빈 바이트 복원
  - 같은 행에 여러 열이 경쟁 → argmax = 가장 빈번한 바이트

## 5. B값 페이지 구조

```
B [0, 63]     25%  R 페이지 (64페이지 × 128열)
B [64, 127]   25%  G 페이지 (64페이지 × 128열)
B [128, 255]  50%  A 정수 존 (128페이지 × 128열)
```

### A 정수 존
- 밝기값이 255를 초과하면 상위 비트를 A존에 분할 저장
- `a_store[a_page][col] = brightness >> 8`
- 캔버스 셀에는 하위 8비트만 유지
- 이미지 렌더링 시 8비트 범위 내 표현 가능

## 6. 스케일링 분석

### 이미지 크기 = f(활성 셀)

```
활성 셀 수 = min(data에 등장하는 (y,x) 조합 수, 512×512)
이미지 크기 ≈ 헤더(24B) + 활성 셀 × 5~7B

텍스트 (43개 고유 문자, 반복구조):
  활성 셀 ≈ 10,496  (고정)
  이미지 ≈ 62~73KB  (원본 크기 무관)

단일 바이트:
  활성 셀 = 1,024   (512 R + 512 G)
  이미지 ≈ 5KB      (원본 크기 무관)

완전 랜덤:
  활성 셀 → 262,144 (모든 셀 활성)
  이미지 ≈ 1.3MB    (상한)
```

### 압축률 = 원본 크기 / 이미지 크기

```
이미지 크기가 고정이므로:
  압축률 ∝ 원본 크기

1MB / 62KB  =   16x
1GB / 73KB  = 14,710x   ← 선형 증가
```

## 8. 한계 및 향후 과제

### 현재 한계
- 비주기적 텍스트: 12% 복원률 (동일 행 내 바이트 경쟁)
- 4MB 미만 소형 데이터: 이미지 오버헤드 > 원본

### 개선 방향
1. **페이지 크기 적응**: 데이터의 자연 주기에 맞춰 페이지 크기 조정
2. **밝기 순위 리스트**: argmax 외에 2위, 3위 열도 활용
3. **멀티 캔버스**: 여러 512×512 이미지를 레이어로 사용
4. **GPU 가속**: 밝기 누산을 GPU 병렬 처리
